# Phase 2 분장 및 협업 방식

## 1) 분장 기준 (인터페이스 중심 분리)

- a 개발자: 검색 파이프라인의 인프라/DB/쿼리 구현 담당
- c 개발자: 임베딩/스코어링/재순위화 로직의 알고리즘/평가 담당
- 접점은 **검색 API 계약**과 **랭킹 결과 포맷**으로 고정

---

## 2) Phase 2 세부 업무 분장

### a 개발자 (백엔드)

- PostgreSQL 쿼리/인덱스 최적화 (pgvector, tsvector, RRF SQL)
- 검색 API 엔드포인트 구현 (/search/hybrid)
- 파라미터 스키마 정의 (top_k, filter, weights 등)
- 성능/부하 테스트 및 캐싱 전략

### c 개발자 (ML)

- 임베딩 모델/전처리 정책 확정
- RRF/가중치 튜닝 전략 수립 및 오프라인 평가
- 품질 측정 지표 정의 (Precision@k, MRR 등)
- 테스트 쿼리셋/골든셋 구축

---

## 3) 협업 방식 (서로의 영역 미수정)

- a 개발자는 **DB/서버 레이어만** 수정
- c 개발자는 **모델/평가 코드만** 수정
- 공통 산출물은 **API 계약 문서**로 합의
  - 요청/응답 스키마
  - weights/RRF_K 파라미터 범위
  - 필터 필드(메타데이터) 정의
- 결과 수렴은 PR 리뷰가 아니라 **계약 문서 기준**으로 통합

---

## 4) 통합 기준 (merge 시 충돌 최소화)

- a 개발자: 서버 레포/DB 스키마 쪽만 변경
- c 개발자: 모델/평가 노트북/튜닝 설정 파일만 변경
- 공통 파일은 **읽기 전용**으로 두고 변경은 대표 1인 승인
